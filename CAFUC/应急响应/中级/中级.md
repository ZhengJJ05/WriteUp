# 应急响应--中级

## 步骤

top查看进程发现watchdog占用cpu过高

<img src=./img/1.png>

使用ps aux | grep watchdog查看进程详情,找到进程路径

<img src=./img/2.png>

直接删除无果,发现程序又重新启动,查找守护进程

<img src=./img/3.png>

查看crontab计划表,发现有定时任务,查看任务详情
查看king后发现是守护进程,并且一直给watchdog添加ai的attr权限

<img src=./img/4.png>

先将计划任务删除,发现不允许删除,为只读权限,查看文件属性
拥有普通读写但是还是不允许修改就考虑attr权限

```bash
lsattr #查看attr权限
chattr -ia #删除attr权限
```

<img src =./img/5.png>

发现chattr居然permission denied,但是我们已经是root用户了
尝试从命令本身入手,查询chattr权限

```bash
find / -name 'chattr' 2>/dev/null
ls -al /usr/bin/chattr
```
发现chattr没有可执行权限,手动添加

<img src=./img/6.png>

```bash
chmod u+x /usr/bin/chattr
```

再回过头来修改attr权限成功,发现可以删除了

<img src=./img/7.png>

删除完计划任务我们去删除守护进程king,相同的步骤,查看普通权限和attr权限
发现king也有attr权限,删除attr权限后,发现可以删除了,停止守护进程后,也不会重启守护进程

<img src=./img/8.png>

现在轮到恶意程序,因为在守护进程中存在systemctl命令对恶意程序的调用,可能存在恶意程序注册服务

使用systemctl对watchdog进行控制

```bash
systemctl status watchdog #查看watchdog状态
systemctl stop watchdog #停止watchdog
systemctl disable watchdog #禁用watchdog
systemctl daemon-reload #重载配置
```

<img src=./img/9.png>

删除恶意文件同样查看普通权限和attr权限,发现也有attr权限,尝试删除

<img src=./img/10.png>

发现报错权限不足,我们查看一下父目录是否有权限,因为父目录是.开头会自动隐藏

```bash
cd /etc
ls -al    #查看所有文件,包括隐藏文件
lsattr -a #查看attr权限,包括隐藏文件
chattr -ia .init #删除attr权限
```

发现父目录普通权限正常但是具有attr的ia权限,删除attr权限后,尝试删除watchdog

<img src=./img/11.png>

<img src=./img/12.png>

成功删除恶意文件

<img src=./img/13.png>

查看top进程,发现watchdog进程已经不存在了,cpu占用恢复正常
使用ps aux | grep 分别查看watchdog,king,发现进程不存在了

<img src=./img/14.png>
